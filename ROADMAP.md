## Director-console hardening punch list (6/11 — Claude red-team + external console review)

**Critical (ship before school #1):**
- **Soft-delete students** — any teacher can hard-delete any student at their school (002 RLS) and scores/notes CASCADE away (001). For a "court of law" record store this is the biggest hole. Add `archived_at` + restore; hard delete founder-only.
- **Multi-teacher category collision** — `nps_get_student_record` (047) and the 038 magic-link views return only the most-recent scoring teacher's categories; rows from other teachers with custom configs score wrong (mismatched ids → 0s, wrong maxes). Score each row against ITS teacher's config.
- ~~Record opens invisible in the director's audit log~~ — ✅ fixed 6/11: `nps_record.*` rows (id-only, no PII payload) now pass the scoped-audit PII filter; audit + links footers rewritten per org shape (NPS director sees the truth, district admins keep the PII-blind copy).
- **DPA / CSDPA registry story** — placing districts can veto unlisted tools regardless of features; longest lead time, start before the 25-email batch (counsel item).

**High (trust + compliance):**
- **Private-note disclosure to teachers** — one sentence at the note composer + in the invite: director and the legal record include "private" notes (they're hidden from parents/other teachers only). First IEP-meeting surprise loses the school.
- **Audit + watermark PDF generation** — `nps_record.print` audit row; "Generated by {email}" in the PDF footer (chain of custody).
- **Record/PDF compliance fields** — date-range picker (districts ask for THEIR quarter, not rolling windows); student ID / placing-district fields on the roster; attendance summary (NPS billing is ADA-based) later.
- **Second school admin** — director can't mint an assistant/successor; one human + one inbox is the school's single point of failure (NPS director turnover is real). Related: shared office@ inboxes → revisit deferred OTP-code/Microsoft-SSO ideas.

**Medium (self-inflicted, quick):**
- **Demo-data containment** — [DEMO] students sit in the real roster: Records tab, usage counts, and every real teacher's dashboard. Add "Wipe demo data" to the director home card + a launch-wizard nudge to wipe before inviting real staff.
- **Director-as-teacher pollution** — `admin_first` teachers row inflates Teachers tab + usage stats (33% noise at a 3-teacher school). Filter admin_first rows from rosters/counts.
- **Launch `finished` flag is per-device** (localStorage) — persist server-side so a new device doesn't replay the wizard.
- **September lockout wave** — sessions die over summer; plan a "welcome back" email with a sign-in button in August.
- **Copy/counter consistency pass** — schedules ("Set" vs "1/1" vs "19 variants"), links, and visibility language differ per tab; staff blurb should reflect the actual link policy (parent links advertised while toggled off).

**External-review items triaged out:** surestep3@proton.me visibility = test-account artifact, not product (real directors see their own email); teacher invites "no approval" is by design (the DIRECTOR is the approval; links are already single-use/24h); the reported deep-link bounce on /admin/upload-schedule has no code path that explains it — needs a repro before chasing.

**Business plumbing (decide, not code):** billing (Stripe at >15 schools?), export/offboarding ("download everything" + retention promise), Supabase PITR + documented recovery promise, mobile eyeball of admin tables at phone width. Backlog niceties: audit search/filter/export + per-student "who viewed" timeline, empty-state CTAs, link-toggle confirmation with affected-link count, "what parents see" preview per link type.

## Working guardrails (current)
- **Reversibility gate** (adopted 6/01): do anything reversible — including backed-up prod data writes; **always snapshot before a prod mutation.** Queue genuine one-way doors (force-push/history rewrite, un-backed-up destructive SQL, bulk ops, real-user emails, infra teardown, auth-config lockout) for an attended, approved moment.
- **Guarded-apply or stage-test migrations** — capture the restore point first; verify after.
- A passing type-check is not a working feature — close the loop in the app.
- `auth.uid()` for attribution, `effective_user_id()` for data access — mixing them caused both 5/29 act-as regressions.
- `onConflict` upserts require a real UNIQUE constraint — verify before trusting.
- RPC responses are not interchangeable with raw row SELECTs even on the same data — watch for shape mismatches.
